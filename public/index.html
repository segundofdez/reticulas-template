Getting Started

Api call

With each API call you must set request headers including an OAuth 2.0 access token. Get an access token by using the OAuth 2.0 client_credentials token grant type with your clientId:secret as your Basic Auth credentials.

Required headers on api call

  • Authorization: Bearer (token)
  • Accept-Language: (one avaliable language code)
  • Accept: application/json

Avaliable languages

  • es (Spanish)
  • en (English)
  • fr (French)
  • eu (Euskera)
  • gl (Galician)
  • ct (Catalan)

Api response

Api response are only available on "application/json" format.

Enviroments

Client_id and secret will be different for each environment, live and sandbox.

For production/live:

  • https://api.monbus.es

For testing propouses/sandbox:

  • https://api.sandbox.monbus.es

On sandbox enviroment we only keep your testing data 12 hours.

Versioning

Only version 1 of the API is available right now. You no need to send an http version header or include a version as part of the url. It's planned change versions of endpoints from customers private area in future.

Authorization

You can obtain an access token for your application by sending a request to the /oauth2/token endpoint. You must authenticate your access token request (using HTTP Basic Auth) with your application credentials obtained form Monbus. The client_id and secret becomes your user-id and password in HTTP Basic Auth.

Endpoint

POST /oauth2/token

Parametres:

Name Type Value Description
grant_type required client_credentials OAuth2 grant type for auth

Request:

curl -X POST https://api.monbus.es/oauth2/token \
                    -H "Authorization: Basic c2FuZGJveDJlNDM0RTVjTjVOdFVZOGcyMTBlUDk4V0p4V3ZMdkJFUmF0MWk0aTdMYUZrSllxcXNIejdCUjlIOTAza214Y2dQbTZzejhQb1U6c2FuZGJveGtvSGdGUDBOdDFtNXo2Sm9QMzNVa1lzcUhnMjBSY1lnUnZVQmVqVWFzNXZFNXhpZUU0eDc0M20yTkpxQjhndDlpRjM4V2FXOWM=" \
                    -H "Accept: application/json" \
                    -H "Content-Type: application/x-www-form-urlencoded" \
                    -d 'grant_type=client_credentials'

Response:

{
                      "access_token": "8dee2542dcac60525ddeb07c8e7481764c9aa3df",
                      "expires_in": 3600,
                      "token_type": "Bearer",
                      "scope": "production"
                    }

Access token validity and expiration

You use access tokens to access all REST API endpoints. These tokens have a finite lifetime and you must write code to detect when an access token expires.

To do this, either:

  • Keep track of the expires_in value in the token response. The value is expressed in seconds.
  • Handle the 401 Unauthorized error response from the API endpoint when an expired token is detected.

Check

The are two endpoints for check if api is alive.

Endpoint

GET /

Request:

curl -X GET https://api.monbus.es \
                    -H "Authorization: Bearer 2734c12345c729d61bda147a39aea86ca7b22c02" \
                    -H "Accept-Language: es" \
                    -H "Accept: application/json"

Response:

{
                      "data": {
                        "api": "Monbus Api",
                          "host": "api.monbus.es",
                          "version": "1"
                      }
                    }

Endpoint

GET /ping

POST /ping

Parametres:

Name Type Value Description

date required Date format Y-m-d Date for test response

Request:

curl -X GET https://api.monbus.es/ping?icmp=test&date=2017-01-15 \
                    -H "Authorization: Bearer 2734c12345c729d61bda147a39aea86ca7b22c02" \
                    -H "Accept-Language: es" \
                    -H "Accept: application/json"
